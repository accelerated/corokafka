29 /*
30  * Kafka-connector specific configs
31  * -----------------------------------------------------------------------
32  * Description: These configuration options are additional to the rdkafka ones. Options
33  * pertaining to a consumer start with 'internal.consumer' and for topics start with 'internal.topic'.
34  * Default values are marked with a * or are in {}.
35  *
36  * internal.consumer.pause.on.start: {'true',*'false'}. The consumer starts in paused state.
37  * User needs to call 'resume' to consume.
38  *
39  * internal.consumer.timeout.ms: {1000} Sets the timeout on any operations requiring a timeout such as poll,
40  * query offsets, etc.
41  *
42  * internal.consumer.poll.timeout.ms: {0} Override the 'internal.consumer.timeout.ms' default setting for polling only.
43  * Set to -1 to disable, in which case 'internal.consumer.timeout.ms' will take precedence. Note that in the case where
44  * 'internal.consumer.poll.strategy=roundrobin', the actual timeout per message will be
45  * 'internal.consumer.poll.timeout.ms/internal.consumer.read.size'. When value is 0, there is no timeout.
46  *
47  * internal.consumer.auto.offset.persist: {*'true','false'}. Enables auto-commit/auto-store inside the corokafka
48  * once each ReceivedMessage has been destroyed.
49  *
50  * internal.consumer.auto.offset.persist.strategy: {'commit',*'store'}. Determines if offsets are committed or stored
51  * locally. This setting only applies when 'internal.consumer.auto.offset.persist=true'. Some rdkafka settings will
52  * be changed according to Note 2 below.
53  *
54  * internal.consumer.commit.exec: {'sync',*'async'}. Dictates if offset commits should be synchronous or asynchronous.
55  *
56  * internal.consumer.commit.num.retries: {MAX_UINT} Sets the number of times to retry committing an offset before giving up.
57  *
58  * internal.consumer.commit.backoff.strategy: {*'linear','exponential'} Backoff strategy when initial commit fails.
59  *
60  * internal.consumer.commit.backoff.interval.ms: {50} Time in ms between retries.
61  *
62  * internal.consumer.commit.max.backoff.ms: {1000} Maximum backoff time for retries. If set, this has higher precedence
63  * than 'internal.consumer.commit.num.retries'.
64  *
65  * internal.consumer.poll.strategy: {'batch',*'roundrobin'}. Determines how messages are read from rdkafka queues.
66  * The 'batch' strategy is to read the entire batch of messages from the main consumer queue. If 'roundrobin'
67  * strategy is selected, messages are read in round-robin fashion from each partition at a time.
68  *
69  * internal.consumer.read.size: {100} Number of messages to read on each poll interval.
70  * See ConnectorConfiguration::setPollInterval() for more details.
71  *
72  * internal.consumer.batch.prefetch: {'true',*'false'} If 'internal.consumer.poll.strategy=batch', start pre-fetching
73  * the next batch while processing the current batch. This increases performance but may cause additional burden
74  * on the broker.
75  *
76  * internal.consumer.receive.callback.thread.range.low: {0} Specifies the lowest thread id on which receive callbacks
77  * will be called. See note 1 below for more details.
78  *
79  * internal.consumer.receive.callback.thread.range.high: {total IO threads} Specifies the highest thread id on which
80  * receive callbacks will be called. See note 1 below for more details. This setting is only valid if
81  * 'internal.consumer.receive.invoke.thread=io'.
82  *
83  * internal.consumer.receive.callback.exec: {'sync',*'async'}. Call the receiver callback asynchronously.
84  * When set to 'async' all received messages from a poll interval (see 'internal.consumer.read.size' above) are
85  * enqueued on the IO threads into the specified thread range (see above) and the receiver callback is called asynchronously,
86  * which means that another poll operation may start immediately. When set to 'sync', the corokafka will deliver each
87  * message synchronously into the specified thread range (see above) and will wait until the entire batch is delivered
88  * to the application, before making another poll operation. In the synchronous case, it's recommended not to do too much
89  * processing inside the receiver callback since it delays the delivery of other messages and to dispatch the message
90  * to some other work queue as soon as possible (see Note 1 for more details).
91  *
92  * internal.consumer.receive.invoke.thread: {*'io','coro'}. Determines where the receiver callback will be invoked from.
93  * In the coroutine case, the execution mode will be forced to sync (i.e. 'internal.consumer.receive.callback.exec=sync')
94  * since 'async' execution has the risk of delivering messages from two consecutive poll intervals out of order.
95  * Receiving messages inside a coroutine could increase performance if the callbacks are non-blocking (or if the block
96  * for very short intervals). When invoking on coroutines, the receiver callback will always be called serially for all
97  * messages inside a particular poll batch.
98  *
99  * internal.consumer.log.level: {'emergency','alert','critical','error','warning','notice',*'info','debug'}.
100  * Sets the log level for this consumer. Note that if the rdkafka 'debug' property is set, internal.producer.log.level
101  * will be automatically adjusted to 'debug'.
102  *
103  * internal.consumer.skip.unknown.headers: {*'true','false'}. If unknown headers are encountered (i.e. for which there
104  * is no registered deserializer), they will be skipped. If set to false, an error will be thrown.
105  *
106  * internal.consumer.auto.throttle: {'true',*'false'}. When enabled, the consumers will be automatically paused/resumed
107  * when throttled by the 'throttle time x multiplier'.
108  *
109  * internal.consumer.auto.throttle.multiplier: {1}. Change this value to pause the consumer by
110  * 'throttle time x multiplier' instead. This only works if 'internal.consumer.auto.throttle=true'.
111  *
112  * internal.consumer.preprocess.messages: {*'true','false'}. Enable the preprocessor callback if it has been registered.
113  * Otherwise it can be enabled/disabled via ConsumerManager::preprocess().
114  *
115  * internal.consumer.preprocess.invoke.thread: {*'io','coro'}. If 'io', run the preprocessor callback on an IO thread.
116  * If 'coro', run it directly on the same coroutine threads where the messages are deserialized. The advantage of invoking
117  * the preprocessor on an IO thread is that any blocking operation can be performed inside the callback. On the other
118  * hand, if no IO operations are needed (or if they are very fast), using coroutine threads will result in better
119  * performance.
120  *
121  * NOTE 1:
122  * -------
123  * Mapping partitions unto IO threads for the receive callback allows for a maximum amount of parallelization
124  * while maintaining arrival order. These two variables allow to manually specify how to distribute 'P' partitions
125  * belonging to a topic unto 'T' IO threads when the receive callback is invoked. The default [low,high] values are
126  * [0,T-1] where T is set to 'numIoThreads' value used in the dispatcher constructor, and the distribution model
127  * is 'partition id % num threads' within the [low,high] range.
128  * This ensures that processing order for all messages is preserved. If a partition reassignment happens,
129  * the range remains unchanged and the new partitions will follow the same distribution mechanism.
130  *
131  * Ex: There are 8 IO threads and 20 partitions assigned to this consumer. We set [low,high] range to
132  * [4,7] which means that the 20 partitions will be mapped unto 4 thread ids {4,5,6,7}. When message from
133  * partition 10 arrives, the thread id on which the receive callback will be invoked can be calculated as
134  * follows: 10 % (7-4+1) + 4 = 6.
135  *
136  * NOTE 2:
137  * -------
138  * Commit strategy:
139  * 'enable.auto.commit=false'
140  * 'enable.auto.offset.store=false'
141  * 'auto.commit.interval.ms=0'
142  * Store strategy:
143  * 'enable.auto.commit=true'
144  * 'enable.auto.offset.store=false'
145  */
